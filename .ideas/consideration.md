# Consideration

## Performance

### Memory Usage

プロジェクションID毎にWorkerプロセスを起動するため、通常は膨大なプロセス数になり得る。
そのため、暫くの間イベントが発生しないプロセスの状態を永続化し、停止することでメモリ使用量を抑えることができる。
しかし、もしイベントの発生通知を受け取り損なった場合、プロセスが起動しないためDB上のスナップショットが古いままになってしまう可能性がある。

### Bottleneck

イベント通知を購読し、プロジェクションに配信し直すRouterは単一のプロセスのため、イベントの発生頻度が高い場合はボトルネックになる可能性がある。
分散環境においても全く同じ数のイベントを購読するため、これは解決にならない。
パーティショニングを行うことで複数のRouterプロセスを起動すれば、イベントの発生頻度が高い場合でもボトルネックを回避できる可能性がある。

WorkerSupervisorはシャーディングにより特定ノードに対し問い合わせを行うが、ノード数だけの問い合わせが発生する。
シャーディングの結果、自分のノードだった場合のみ問い合わせを行うか、イベントをハッシュしてキャッシュし、既知のイベントの場合は無視するようにすることで、問い合わせ数を減らすことができる可能性がある。
後者の場合は冗長性を確保できるため、イベントの発生通知を受け取り損なう可能性が減る。
